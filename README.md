## Цель проекта

Разработать веб-приложение, которое позволяет:

1. Собирать данные о недвижимости (парсинг и ручной ввод) с расширенными характеристиками.
2. Хранить данные в PostgreSQL.
3. Предоставлять REST API для объектов, пользователей и избранного.
4. Прогнозировать цену на основе ML-модели (отдельный микросервис на Python).
5. Визуализировать прогнозы и региональную аналитику (дашборды, графики).
6. Разворачивать все компоненты в Docker-контейнерах.

## Как работает система

### 1. Просмотр объектов

Пользователь открывает сайт → Frontend запрашивает `GET /api/properties` → Go читает PostgreSQL → список квартир с фильтром по городу.

### 2. Регистрация и избранное

- `POST /api/register` — пользователь в БД, пароль хешируется (bcrypt).
- `POST /api/login` — выдаётся JWT на 24 часа.
- Избранное: `POST/DELETE /api/favorites/{id}` с заголовком `Authorization: Bearer <token>`.

### 3. Прогноз цены

1. Пользователь заполняет форму (город, площадь, комнаты, …).
2. Frontend → `POST /api/predict` → **Go Backend**.
3. Go проксирует запрос → `POST /predict` → **ML Service**.
4. ML загружает модель Random Forest, предсказывает цену.
5. ML считает аналитику: средняя по городу, % к рынку, прогноз на 12 месяцев, текстовый тренд.
6. Ответ возвращается на фронт → **дашборд с графиками** (Recharts).

### 4. Сбор данных парсером

`scheduler.py` раз в 24 часа запускает `parser.py` → для каждого из 26 городов вызывается `cianparser` (если установлен) → INSERT в PostgreSQL с дедупликацией по `source_url`.

### 5. Переобучение ML

- При старте ML-сервиса.
- Каждые 100 новых объектов в БД.
- После ручного добавления объекта (Go вызывает `/retrain` в фоне).
- Вручную: `POST http://localhost:8000/retrain`.

## Модель ML

Раздел описывает алгоритм прогнозирования цены. Реализация: `ml-service/app/model_trainer.py`, `ml-service/app/analytics.py`.

### Тип задачи

**Регрессия** — по характеристикам квартиры предсказывается непрерывная величина: **цена в рублях**.

- **Вход (X):** признаки объекта (площадь, город, этаж и т.д.).
- **Выход (y):** `price` — фактическая цена при обучении; предсказанная цена при инференсе.

### Алгоритм: Random Forest Regressor

Используется **Random Forest** из библиотеки **scikit-learn** — ансамбль решающих деревьев. Итоговая цена — среднее предсказаний всех деревьев.

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `n_estimators` | 200 | Число деревьев в ансамбле |
| `max_depth` | 12 | Ограничение глубины (защита от переобучения) |
| `random_state` | 42 | Воспроизводимость результатов |
| `n_jobs` | -1 | Параллельное обучение на всех ядрах CPU |

**Почему Random Forest (а не линейная регрессия):**

- работает с **категориальными** признаками (город, район, тип дома);
- улавливает **нелинейные** зависимости (цена ↔ площадь ↔ регион);
- устойчив к выбросам в данных;
- не требует масштабирования числовых признаков.

### Признаки (features)

#### Числовые

| Признак | Описание |
|---------|----------|
| `area` | Площадь, м² |
| `rooms` | Количество комнат |
| `floor` | Этаж |
| `total_floors` | Этажей в доме |
| `year_built` | Год постройки |

#### Категориальные (one-hot encoding)

| Признак | Примеры |
|---------|---------|
| `city` | Москва, Казань, Новосибирск |
| `district` | Район города |
| `metro` | Станция метро |
| `building_type` | кирпичный, панельный, монолитный |
| `developer` | Застройщик |
| `repair_type` | Ремонт квартиры: евроремонт, косметический |
| `building_repair_type` | Ремонт дома: капитальный, косметический |

**Не используются в модели:** `address`, `source_url` — слишком уникальны, не несут обобщающей информации.

#### Обработка пропусков

- Категориальные поля: пустое значение → `"unknown"`.
- Числовые поля: пустое значение → **медиана** по обучающей выборке (сохраняется вместе с моделью в `.joblib`).

### Предобработка и pipeline

Все шаги объединены в **scikit-learn Pipeline**:

```
Входные признаки
       ↓
ColumnTransformer
  ├── числовые столбцы → без преобразования
  └── категориальные → OneHotEncoder (handle_unknown="ignore")
       ↓
RandomForestRegressor
       ↓
log₁₊(price)  →  при прогнозе: expm1()  →  цена в ₽
```

`OneHotEncoder` превращает категорию «Казань» в бинарный вектор. Параметр `handle_unknown="ignore"` позволяет обрабатывать город, не встречавшийся при обучении.

### Целевая переменная: логарифм цены

Модель обучается на **`log1p(price)`**, а не на сырой цене:

```python
y = np.log1p(df["price"])      # обучение
price = np.expm1(prediction)   # прогноз
```

**Причина:** цены на недвижимость сильно разбросаны (от 3 до 30+ млн ₽). Логарифм сжимает масштаб и стабилизирует обучение.

### Данные для обучения

При каждом обучении ML-сервис загружает **все записи** из таблицы `properties` в PostgreSQL:

```sql
SELECT area, rooms, city, district, metro, floor, total_floors,
       building_type, year_built, developer, repair_type, building_repair_type, price
FROM properties
WHERE price > 0 AND area > 0 AND rooms > 0
```

- Минимум записей для обучения: **5** (`MIN_TRAIN_SAMPLES`).
- Рекомендуется для качественной модели: **от 1000+** объектов (по ТЗ).

### Процесс обучения

1. Загрузка данных из PostgreSQL.
2. Очистка: удаление строк без цены, площади, города.
3. Разбиение **80/20** (train/test), если записей ≥ 10; иначе обучение на всей выборке.
4. Обучение pipeline (препроцессор + Random Forest).
5. Расчёт метрик на тестовой выборке.
6. Сохранение в `models/price_model.joblib`:
   - обученный `pipeline`;
   - словарь `defaults` (медианы числовых признаков).

### Метрики качества

Оценка производится **в рублях** (после обратного преобразования `expm1`):

| Метрика | Формула (смысл) | Интерпретация |
|---------|-----------------|---------------|
| **MAE** | Средняя абсолютная ошибка | «В среднем ошибаемся на X ₽» |
| **RMSE** | Корень из средней квадратичной ошибки | Большие промахи штрафуются сильнее |
| **R²** | Коэффициент детерминации | Доля объяснённой дисперсии (1.0 = идеально) |

Метрики возвращаются в ответе `POST /retrain` и логируются при старте сервиса.

### Когда модель переобучается

| Событие | Описание |
|---------|----------|
| Старт `ml-service` | Автоматическое обучение в `lifespan` FastAPI |
| Каждые **100** новых объектов | Счётчик записей в БД (`RETRAIN_EVERY_N`) |
| Добавление объекта через API | Go-бекенд вызывает `/retrain` в фоне |
| Вручную | `POST http://localhost:8000/retrain` |

При переобучении модель заново читает **все** актуальные данные из БД.

### Процесс прогноза (инференс)

1. Пользователь отправляет JSON с признаками квартиры.
2. Go-бекенд проксирует запрос в `POST /predict` ML-сервиса.
3. Загружается `price_model.joblib` (или обучение «на лету», если файла нет).
4. Заполняются пропуски, применяется обученный pipeline.
5. Random Forest выдаёт `log_price` → `expm1` → **predicted_price**.
6. Модуль `analytics.py` добавляет:
   - среднюю цену в городе из БД;
   - отклонение от рынка, %;
   - текстовый тренд;
   - прогноз цены на **12 месяцев**.

### ML vs аналитика (важно для защиты)

Не все числа на дашборде — результат Random Forest:

| Компонент | Метод | ML? |
|-----------|--------|-----|
| **Предсказанная цена** | Random Forest + корректировки (`price_adjustments.py`: район, ремонт, застройщик, год) | ✅ + правила |
| **Цена за м²** | predicted_price / area | ❌ формула |
| **Средняя по городу** | AVG из PostgreSQL | ❌ статистика |
| **% к рынку** | сравнение с средним | ❌ статистика |
| **Текст тренда** | справочник + правила | ❌ эвристика |
| **График на 12 мес.** | сложный процент от годового % региона | ❌ формула |

Годовой рост по регионам задан в `ml-service/app/regions_config.py` (например, Казань — 6.2%/год, Москва — 4.5%/год).

### Схема ML-процесса

```
┌──────────────────┐
│   PostgreSQL     │
│   properties     │
└────────┬─────────┘
         │ SELECT (обучение)
         ▼
┌──────────────────┐     train/test 80/20

│ 200 деревьев     │
└────────┬─────────┘
         │ predict (инференс)
         ▼
┌──────────────────┐
│  Цена в ₽ +      │
│  аналитика       │ ──► JSON → Frontend (графики)
└──────────────────┘
```

### Ограничения модели

1. **Объём данных** — на демо-выборке (~40–50 объектов) метрики условны; для продакшена нужны тысячи записей с парсера.
2. **Нет временных рядов** — модель не знает историю изменения цен; график на 12 месяцев строится по справочному % роста региона, а не по ML.
3. **Новые категории** — неизвестный город/район кодируется как `unknown`, прогноз менее точен.
4. **Качество входных данных** — ошибки парсера или ручного ввода напрямую влияют на обучение.

- **Библиотека:** [cianparser](https://pypi.org/project/cianparser/) (опционально).
- **Расписание:** раз в 24 часа (`PARSER_INTERVAL_HOURS` в docker-compose).
- **Города:** 26 регионов (см. ниже).
- **Дедупликация:** по полю `source_url`.
- **Без cianparser:** добавляются демо-записи для наполнения БД.

Запуск вручную:

```bash
docker compose exec parser python parser.py
```

## Веб-интерфейс

URL: **http://localhost:5173**

| Страница | Описание |
|----------|----------|
| Объекты | Список квартир, фильтр по городу, избранное |
| Карточка объекта | Все поля, ссылка на объявление |
| Прогноз и аналитика | Форма ML + дашборд (графики Recharts) |
| Добавить | Ручной ввод (JWT) |
| Избранное | Сохранённые объекты |
| Вход / Регистрация | Авторизация |

> `http://localhost:8080/api` — это JSON API для программ, не веб-страница. Пользовательский интерфейс — только на порту **5173**.

## Обновление данных

| Источник | Частота | Описание |
|----------|---------|----------|
| **Парсер** | 24 часа (настраивается) | Новые объявления по 26 городам |
| **Ручной ввод** | Сразу | Авторизованный пользователь |
| **ML-модель** | Каждые 100 объектов | Переобучение на актуальной БД |
| **Первый запуск** | Один раз | SQL-сиды в `db/init.sql` |

Изменить интервал парсера в `docker-compose.yml`:

```yaml
PARSER_INTERVAL_HOURS: "12"
```

## Быстрый старт (Docker)

### Требования

- Docker Desktop
- Git

### Запуск

```bash
git clone https://github.com/dratbo/property-price-predictor.git
cd property-price-predictor
cp .env.example .env
docker compose up --build
```

Подождите 1–2 минуты (обучение ML на старте).

### Адреса

| Сервис | URL |
|--------|-----|
| **Веб-приложение** | http://localhost:5173 |
| **REST API** | http://localhost:8080/api |
| **ML health** | http://localhost:8000/health |
| **PostgreSQL** | localhost:5433 |

### Остановка

```bash
docker compose down
```

Полная очистка данных БД:

```bash
docker compose down -v
```

## Локальная разработка

### PostgreSQL

```bash
docker run -d --name property-db \
  -e POSTGRES_USER=property \
  -e POSTGRES_PASSWORD=property \
  -e POSTGRES_DB=property_db \
  -p 5433:5432 \
  -v ./db/init.sql:/docker-entrypoint-initdb.d/01_init.sql \
  -v ./db/seed_extended_regions.sql:/docker-entrypoint-initdb.d/02_seed_extended.sql \
  postgres:15-alpine
```

### ML-сервис

```bash
cd ml-service
pip install -r requirements.txt
set DATABASE_URL=postgres://property:property@localhost:5433/property_db
uvicorn app.main:app --reload --port 8000
```

### Backend

```bash
cd backend
set DATABASE_URL=postgres://property:property@localhost:5433/property_db?sslmode=disable
set ML_SERVICE_URL=http://localhost:8000
set JWT_SECRET=your-secret-key
go run ./cmd/api
```

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Откроется http://localhost:5173 (прокси `/api` → backend настроен в `vite.config.js`).

## Демонстрация для защиты

1. `docker compose up --build` — поднять все сервисы.
2. Открыть http://localhost:5173 — список объектов, фильтр по городу.
3. Карточка объекта — расширенные поля.
4. Регистрация → добавить объект → появился в списке.
5. Избранное — добавить/удалить.
6. **Прогноз и аналитика** — Казань, 50 м², 2 комнаты → графики, KPI, тренд на 12 месяцев.
7. Показать http://localhost:8000/health — ML работает.
